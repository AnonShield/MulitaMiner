   - Extract from HEADING TEXT ONLY (2-15 words before vulnerability details)
   - ❌ NEVER: Description text like "The scanner detected..." or long multi-sentence phrases
   - ✅ ALWAYS: Exact heading text as it appears in PDF

2. OPENVAS ACCEPTANCE PATTERNS (PERMISSIVE):
   ✅ ACCEPT: "Abnormal HTTP Response Code" - legitimate OpenVAS finding
   ✅ ACCEPT: "Abnormal HTTP Response Code Instances (N)" - legitimate OpenVAS finding
   ✅ ACCEPT: "Access to .htaccess file" - legitimate finding in network scans
   ✅ ACCEPT: "Access to .htpasswd file" - legitimate finding in network scans
   ✅ ACCEPT: "SMTP too long line" - valid network detection
   ✅ ACCEPT: "SSL/TLS Certificate" vulnerabilities
   ✅ ACCEPT: Port and service detection findings
   ❌ REJECT: Only if clearly NOT in PDF heading

3. EXTRACT NAMES EXACTLY FROM PDF HEADINGS
   - Preserve original text, capitalization, punctuation
   - Do NOT paraphrase or rearrange
   - CRITICAL: If heading contains "Instances (N)" suffix, REMOVE IT from the name

   - Extract instance numbers when present: "Port Scan Instances (5)" → use "Port Scan" as name

   - Consolidate all instances into single object with complete data

CAIS FIELD MAPPING - OPENVAS NETWORKS
═══════════════════════════════════════════════════════════════════════════════════

ASSET INFORMATION (OpenVAS):
📍 asset.name → Scan target name or IP address
📍 asset.display_fqdn → Hostname if available (otherwise same as IP)
📍 asset.display_ipv4_address → IPv4 address from target or scan header
📍 asset.host_name → Hostname portion (if FQDN available)
📍 asset.operating_system → OS detected from scan (if mentioned)
📍 asset.system_type = "Network Service" (always for OpenVAS)

DEFINITION FIELDS - CORE:
🔸 definition.name → Vulnerability title from heading (remove "Instances (N)" suffix)
🔸 name_consolidated → Same as definition.name for deduplication logic
    - For "Abnormal HTTP Response Code Instances (2)" → name_consolidated = "Abnormal HTTP Response Code"
    - For "Port Configuration Issue" → name_consolidated = "Port Configuration Issue"
    - Use this field to group and consolidate duplicate vulnerabilities
🔸 definition.severity → Severity level (CRITICAL|HIGH|MEDIUM|LOW|LOG)
🔸 definition.description → Description text from vulnerability sections
🔸 definition.solution → Solution steps from vulnerability sections
🔸 definition.id → Plugin ID in OID format (e.g., "1.3.6.1.4.1.25623.1.0.11270")

DEFINITION FIELDS - DETAILED (extract if available):
🔍 definition.cve → CVE identifier (CVE-YYYY-XXXXX format) from references or description
🔍 definition.cwe → CWE identifier (CWE-XXXXX format) from references or description
🔍 definition.family → Vulnerability family based on network service type:
    • "SMTP" for email service vulnerabilities
    • "SSL/TLS" for encryption and certificate issues
    • "HTTP" for web service vulnerabilities
    • "DNS" for DNS service issues
    • "SSH" for SSH service vulnerabilities
    • "FTP" for FTP service issues
    • "Port Scan" for port discovery findings
🔍 definition.type → Specific vulnerability type:
    • "Denial of Service" for DoS vulnerabilities
    • "Information Disclosure" for data exposure
    • "Configuration Issue" for service misconfigurations
    • "Certificate Issue" for SSL/TLS certificate problems
🔍 definition.references → URLs from "References" sections
🔍 definition.synopsis → First sentence or brief summary from description
🔍 definition.cvss3.base_score → CVSS v3 base score (numeric value)
🔍 definition.cvss3.base_vector → CVSS v3 vector string (CVSS:3.1/AV:N/AC:L/...)
🔍 definition.cvss2.base_score → CVSS v2 score if available
🔍 definition.cvss2.base_vector → CVSS v2 vector if available
🔍 definition.see_also → Additional reference URLs or documentation links
🔍 definition.plugin_published → Plugin creation/publication date if mentioned
🔍 definition.vulnerability_published → Original CVE disclosure date if available
🔍 definition.patch_published → Vendor patch release date if mentioned

DETECTION FIELDS:
🎯 output → Technical proof/output from network scan
🎯 port → Port number (numeric, specific TCP/UDP port)
🎯 protocol → Protocol type (tcp or udp)

SCAN METADATA:
🔬 scan.id → Scan identifier or report ID if available
🔬 scan.target → Scan target (IP or hostname)

TIMING FIELDS (if available):
⏰ first_observed → Detection date if available
⏰ last_seen → Last detection date
⏰ age_in_days → Days since first observation (calculated if dates available)

STATE & TRACKING:
📈 severity → Same as definition.severity (for consistency)
📈 state → Vulnerability state ("Open", "Fixed", "Verified", etc.) if mentioned
📈 id → Unique identifier for this vulnerability instance
📈 source → Always "OPENVAS" for OpenVAS reports

OPENVAS EXTRACTION WORKFLOW:
1️⃣ Scan report for vulnerability headings and NVT blocks
2️⃣ For each heading, extract vulnerability name (remove "Instances (N)" if present)
3️⃣ Extract technical details from vulnerability sections
4️⃣ Fill CAIS object with network service data:
    • Use heading text as definition.name
    • Fill description/solution from vulnerability content
    • Extract port/protocol from service information
    • Extract CVSS scores from risk information
5️⃣ Set asset.system_type = "Network Service"
6️⃣ Extract asset information from scan target (IP/hostname)
7️⃣ Set source = "OPENVAS"
