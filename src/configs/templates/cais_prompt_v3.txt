EXTRACTION LOGIC (SCANNER-AWARE)
═══════════════════════════════════════════════════════════════════════════════════

Apply extraction rules adapted for each scanner type:

1. NEVER EXTRACT DESCRIPTION TEXT AS NAMES
   - Extract from HEADING TEXT ONLY (2-15 words before "VULNERABILITY [SEVERITY] PLUGIN ID")
   - ❌ NEVER: Description text like "The scanner detected..." or long multi-sentence phrases
   - ✅ ALWAYS: Exact heading text as it appears in PDF

2. SCANNER-SPECIFIC REJECTION PATTERNS:
   
   FOR TENABLE WAS (strict - reject web-specific fabrications):
   ❌ "Access to .htaccess file" - only valid if appears in heading
   ❌ "Access to .htpasswd file" - only valid if appears in heading
   ❌ "Access to .htaccess file denied" - fabricated (OUTPUT/PROOF section)
   ❌ "Access to .htpasswd file denied" - fabricated (OUTPUT/PROOF section)
   ❌ "Abnormal HTTP Response Code" - fabricated name
   ✅ ACCEPT: "Abnormal HTTP Response Code Instances (N)" if exact heading
   
   FOR OPENVAS (permissive - allow legitimate network findings):
   ✅ ACCEPT: "Abnormal HTTP Response Code" - legitimate OpenVAS finding
   ✅ ACCEPT: "Abnormal HTTP Response Code Instances (N)" - legitimate OpenVAS finding
   ✅ ACCEPT: "Access to .htaccess file" - legitimate finding in network scans
   ✅ ACCEPT: "Access to .htpasswd file" - legitimate finding in network scans
   ✅ ACCEPT: "Abnormal HTTP Response Code" - valid in HTTP detection context
   ❌ REJECT: Only if clearly NOT in PDF heading

3. EXTRACT NAMES EXACTLY FROM PDF HEADINGS
   - Preserve original text, capitalization, punctuation
   - Do NOT paraphrase or rearrange
   - CRITICAL: If heading contains "Instances (N)" suffix, REMOVE IT from the name
   - Example: "PHP Unsupported Version Instances (14)" → extract as "PHP Unsupported Version"
   - Example: "Apache 2.4.x < 2.4.41 Multiple Vulnerabilities Instances (1)" → extract as "Apache 2.4.x < 2.4.41 Multiple Vulnerabilities"

4. NAME CONSOLIDATION
   - Merge duplicate names with different instance counts
   - Extract instance numbers when present: "Missing Cache Control Instances (14)" → use "Missing Cache Control" as name
   - TENABLE WAS: Typically has two headings per vuln (with and without "Instances") → always use base name (without "Instances")
   - OPENVAS: May have single heading or heading with instances → always use base name (without "Instances")

5. INSTANCES HANDLING
   - Extract ONLY ONCE for each unique vulnerability
   - If http_info + identification both present → consolidate into single object

6. FINAL VERIFICATION
   - Validate all names came from actual PDF headings
   - For OpenVAS: less strict about common patterns
   - For Tenable WAS: strict validation to avoid fabrications

CRITICAL: Tenable WAS has TWO types of vulnerabilities that must be extracted and consolidated:

TYPE 1: INSTANCES VULNERABILITIES (URLs and HTTP Info)
🔸 Name pattern: "Vulnerability Name Instances (N)"
🔸 Contains INSTANCE blocks with URLs and HTTP technical data

TYPE 2: BASE VULNERABILITIES (Descriptions and Solutions)  
🔸 Name pattern: "Vulnerability Name" (no "Instances" suffix)
🔸 Contains Description, Solution, Plugin Details sections
🔸 Contains Risk Information with CVSS scores and references
🔸 Focus: description, solution, plugin, cvss, references arrays

CONSOLIDATION STRATEGY:
🔹 ALWAYS use BASE name (without "Instances (N)") as definition.name
🔹 Merge data from both INSTANCES and BASE into single CAIS object
🔹 Fill identification/http_info from INSTANCES sections
🔹 Fill description/solution/plugin/cvss from BASE sections
🔹 Result: One comprehensive vulnerability entry per unique base name

TENABLE WAS NAME EXTRACTION RULES:
📋 Look for "VULNERABILITY [SEVERITY] PLUGIN ID [number]" headers
📋 Extract vulnerability name from text BEFORE the VULNERABILITY keyword
📋 Strip "Instances (N)" suffix for consolidation: "PHP Unsupported Version Instances (14)" → "PHP Unsupported Version"
📋 Validate name appears in actual PDF headings, not description text
📋 Reject technical evidence like "Access to .htaccess file denied" (use in output field instead)

EXAMPLES OF VALID TENABLE WAS EXTRACTION:
✅ "Cross-Site Scripting (XSS) - Stored"
✅ "SQL Injection"  
✅ "PHP Unsupported Version" (from both base and instances sections)
✅ "Missing HTTP Strict Transport Security Policy"
✅ "Directory Listing" (from both base and instances sections)
✅ "Apache 2.4.x < 2.4.41 Multiple Vulnerabilities"
✅ "Cookie Without Secure Flag Detected"

EXAMPLES OF INVALID TENABLE WAS EXTRACTION:
❌ "Access to .htaccess file denied" (technical evidence, not vulnerability name)
❌ "Response code '200' triggered by request" (technical output, not vulnerability name)
❌ "Affected URL: https://example.com" (evidence line, not vulnerability name)
❌ "The scanner detected..." (description text, not heading)

ASSET INFORMATION (Tenable WAS):
📍 asset.name → Application/target name from scan header or main URL domain
📍 asset.display_fqdn → Domain extracted from URLs (e.g., juice-shop.herokuapp.com from https://juice-shop.herokuapp.com/path)
📍 asset.display_ipv4_address → IP if resolved in report (usually null for web apps)
📍 asset.host_name → Hostname portion of FQDN (e.g., "juice-shop" from "juice-shop.herokuapp.com")
📍 asset.operating_system → null (typically not available for web applications)
📍 asset.system_type = "Web Application" (always for Tenable WAS scans)

DEFINITION FIELDS - CORE (consolidated from both INSTANCES and BASE):
🔸 definition.name → Base vulnerability name (without "Instances (N)" suffix)
🔸 name_consolidated → Same as definition.name for deduplication logic
🔸 definition.severity → Severity level from VULNERABILITY header (CRITICAL|HIGH|MEDIUM|LOW|LOG)
🔸 definition.description → Description text from BASE vulnerability sections
🔸 definition.solution → Solution steps from BASE vulnerability sections
🔸 definition.id → Plugin ID from VULNERABILITY header (e.g., "12345")

DEFINITION FIELDS - DETAILED (extract if available from BASE sections):
🔍 definition.cve → CVE identifier (CVE-YYYY-XXXXX format) from references or description
🔍 definition.cwe → CWE identifier (CWE-XXXXX format) from references or description  
🔍 definition.family → Issue category based on vulnerability type:
    • "Cross-Site Scripting" for XSS vulnerabilities
    • "SQL Injection" for SQL injection vulnerabilities
    • "Information Disclosure" for data exposure issues
    • "Authentication" for login/auth issues
    • "Authorization" for access control issues
    • "Configuration" for server/app misconfigurations
    • "Encryption" for crypto/TLS issues
🔍 definition.type → Specific vulnerability type matching definition.family
🔍 definition.references → URLs from "See Also", "References", or "Reference Information" sections
🔍 definition.synopsis → First sentence or brief summary from description
🔍 definition.cvss3.base_score → CVSS v3 base score (numeric value)
🔍 definition.cvss3.base_vector → CVSS v3 vector string (CVSS:3.1/AV:N/AC:L/...)
🔍 definition.cvss2.base_score → CVSS v2 score if available (less common)
🔍 definition.cvss2.base_vector → CVSS v2 vector if available
🔍 definition.see_also → Additional reference URLs or documentation links
🔍 definition.plugin_published → Plugin creation/publication date if mentioned
🔍 definition.vulnerability_published → Original CVE disclosure date if available
🔍 definition.patch_published → Vendor patch release date if mentioned

DETECTION FIELDS (consolidated from INSTANCES sections):
🎯 output → Technical proof/evidence from INSTANCES findings and HTTP responses
🎯 port → Port number extracted from URLs (443 for https://, 80 for http://, or custom port)
🎯 protocol → Protocol scheme from URLs ("https" or "http")

EVIDENCE FIELDS (from INSTANCES sections):
📊 identification → Array of ALL affected URLs from INSTANCE blocks and Identification sections
📊 http_info → Array of HTTP technical details:
    • REQUEST method and URL (e.g., "GET https://example.com/path HTTP/1.1")
    • REQUEST HEADERS (Accept, User-Agent, Cookie, etc.)
    • RESPONSE HEADERS (Server, Content-Type, Set-Cookie, etc.)
    • RESPONSE STATUS (e.g., "HTTP/1.1 200 OK")

SCAN METADATA:
🔬 scan.id → Scan identifier or report ID if available
🔬 scan.target → Main target URL or application URL being scanned

TIMING FIELDS (if available):
⏰ first_observed → First detection date if mentioned in scan metadata
⏰ last_seen → Last detection date or scan completion date
⏰ age_in_days → Days since first observation (calculated if dates available)

STATE & TRACKING:
📈 severity → Same as definition.severity (for consistency)
📈 state → Vulnerability state ("Open", "Fixed", "Verified", etc.) if mentioned
📈 id → Unique identifier for this vulnerability instance
📈 source → Always "TENABLEWAS" for Tenable WAS reports

HOW TO APPLY THE CORRECT SECTION
═══════════════════════════════════════════════════════════════════════════════════

SECTION SELECTION LOGIC:

1. DETECT SCANNER TYPE FROM PDF CONTENT:
   
   Use "CAIS FIELD MAPPING - TENABLE WAS" section if:
   ✓ PDF contains web application URLs (e.g., https://juice-shop.herokuapp.com)
   ✓ Vulnerabilities are web-specific (XSS, SQL Injection, CSRF, etc)
   ✓ Scanning target is a domain/application URL
   ✓ STRICT EXTRACTION: Reject fabricated names like "Abnormal HTTP Response Code"
   ✓ asset.system_type = "Web Application"
   ✓ asset.display_ipv4_address = null
   ✓ port from URL (443 for https, 80 for http, custom if specified)
   ✓ protocol = http or https
   
   Use "CAIS FIELD MAPPING - OPENVAS" section if:
   ✓ PDF contains IP addresses (e.g., 192.168.1.10)
   ✓ Vulnerabilities are network-specific (SMTP, SSH, SSL/TLS, Open Ports, etc)
   ✓ Scanning target is an IP or hostname
   ✓ PERMISSIVE EXTRACTION: Accept patterns like "Abnormal HTTP", file access findings
   ✓ Extract ALL valid findings even if names seem unusual
   ✓ asset.system_type = "Network Service"
   ✓ asset.display_ipv4_address = actual IP value
   ✓ port = specific TCP/UDP number
   ✓ protocol = tcp or udp

2. REJECTION LOGIC BY SCANNER:

   TENABLE WAS (Web Apps):
   ❌ "Abnormal HTTP Response Code" ← NOT in web app headings, describe in output instead
   ❌ "Access to .htaccess file denied" ← fabricated from OUTPUT, not heading
   ❌ "Access to .htpasswd file denied" ← fabricated from OUTPUT, not heading
   ✅ "Abnormal HTTP Response Code Instances (5)" ← if exact heading in PDF
   
   OPENVAS (Network):
   ✅ "Abnormal HTTP Response Code" ← legitimate HTTP anomaly finding
   ✅ "Abnormal HTTP Response Code Instances (N)" ← legitimate network finding
   ✅ "Access to .htaccess file" ← legitimate file access finding
   ✅ "Access to .htpasswd file" ← legitimate file access finding
   ❌ Only reject if CLEARLY not in PDF heading text

3. APPLY ONCE PER VULNERABILITY:
   - Extract each vulnerability only once using the appropriate section
   - Do NOT create duplicate entries if vulnerability appears in both contexts
   - Choose the PRIMARY context based on scanning method

4. VALIDATE BEFORE RETURNING:
   - Tenable WAS: Strictly validate names, reject suspicious patterns
   - OpenVAS: Accept more patterns, focus on legitimate network findings
   - Both: definition.severity, definition.cwe/cve where available
   - All returned objects use CAIS field names (with dots)
