EXTRACTION RULES
═══════════════════════════════════════════════════════════════════════════════════

1. ABSOLUTE REJECTION PATTERNS (NEVER EXTRACT):
   ❌ "Access to .htpasswd file denied" ← Fabricated OUTPUT/PROOF text
   ❌ "Access to .htaccess file denied" ← Fabricated OUTPUT/PROOF text
   ❌ "Abnormal HTTP Response Code" ← Fabricated pattern
   ❌ Anything starting with "The", "No ", "Page has no"
   ❌ Anything containing only "File", "Access to", "HTTP status codes"
   ✅ When unsure → REJECT IT

2. EXTRACT NAMES EXACTLY FROM PDF HEADINGS
   ✅ Use heading text as "definition.name" - short text (2-15 words) before "PLUGIN ID"
   ✅ Preserve original capitalization and wording
   ❌ Do NOT paraphrase

3. FIELD MAPPING FROM REPORT SECTIONS:
   
   ASSET INFORMATION (from scan metadata):
   - Extract target URL/domain → "asset.display_fqdn"
   - Extract target IP → "asset.display_ipv4_address"
   - If multiple IPs → "asset.ipv4_addresses" (comma-separated)
   - Detect OS from report → "asset.operating_system"
   - Type = "web application", "server", "network device", etc
   
   VULNERABILITY DEFINITION:
   - Name from heading → "definition.name"
   - Summary/Description section → "definition.description"
   - Solution section → "definition.solution"
   - References section → "definition.references" (comma-separated)
   - Plugin ID → "definition.id"
   - CVSS scores → "definition.cvss[2|3|4].base_score" (as numbers, not strings)
   - CVE identifier → "definition.cve"
   - CWE identifier → "definition.cwe"
   - Severity from report → "definition.severity"
   - Extract family/type information → "definition.family", "definition.type"
   
   DETECTION:
   - Output/Proof/Technical Details section → "output" (full text or technical evidence)
   - Port number → "port" (as number)
   - Protocol → "protocol" (http, https, tcp, udp, etc)
   - Severity level → "severity"
   - Current status → "state" (Open, Fixed, etc)

4. DATA TYPE REQUIREMENTS:
   - port, age_in_days, time_to_fix, vuln_age: MUST be numbers or null, NOT strings
   - CVSS scores: MUST be numbers (e.g., 7.5) or null, NOT "7.5" string or array
   - boolean fields: use true/false or null, NEVER "true"/"false" strings
   - dates: ISO 8601 format "YYYY-MM-DD" or null
   - all other fields: strings or null

5. NULL HANDLING:
   - If field cannot be found/extracted → null (not empty string, not 0)
   - If value is empty → null
   - If value is "N/A", "-", "Unknown" → null

6. CONSOLIDATION:
   - One JSON object per vulnerability instance found
   - Each object must have ALL 87 fields
   - Return as JSON array: [{ field1, field2, ... }, { ... }, ...]

SCANNER-SPECIFIC RULES
═══════════════════════════════════════════════════════════════════════════════════

OPENVAS REPORTS:
- NVT field → "definition.name"
- OID → "definition.id"
- Summary → "definition.description"
- Solution → "definition.solution"
- CVSS field → "definition.cvss3.base_score" (extract number)
- Port from report → "port"
- Protocol typically "tcp"
- No CVSS2/CVSS4 available → leave as null

TENABLE WAS REPORTS:
- Vulnerability name → "definition.name"
- Plugin ID → "definition.id"
- Description → "definition.description"
- Solution → "definition.solution"
- Plugin Details section → Extract CVSS versions and scores
- Family → "definition.family"
- Port → "port" (from URL or explicitly stated)
- Protocol usually "http" or "https"

NESSUS REPORTS:
- Plugin Name → "definition.name"
- Plugin ID → "definition.id"
- Description → "definition.description"
- Solution → "definition.solution"
- CVSS Base Score → "definition.cvss3.base_score"
- Risk Factor → "definition.severity"
- Port → "port"
- Protocol → "protocol"
